# Micro Design Document: Rotations (Galois Automorphisms)

## 1. Introduction

Homomorphic rotations are a powerful feature in CKKS, allowing the slots of an encrypted vector to be cyclically shifted. This is analogous to `numpy.roll` on a plaintext vector but performed on the ciphertext. Rotations are essential for many applications, including convolutions (used in image processing and neural networks), polynomial evaluation, and some bootstrapping techniques.

In CKKS, rotations are achieved by applying Galois automorphisms to the ciphertext, followed by a key-switching procedure.

## 2. Purpose

*   To define how cyclic shifts (rotations) are performed on the vector of values encrypted in a `Ciphertext`.
*   To specify the role of Galois keys in this process.
*   To outline the key-switching step that is part of a rotation.

## 3. The `Evaluator` Class Method

Rotations will be methods within the `Evaluator` (or main `CKKS`) class.

*   **`rotate(self, ct: Ciphertext, steps: int, galois_key: GaloisKey) -> Ciphertext`**:
    *   Input:
        *   `ct: Ciphertext`: The ciphertext to be rotated.
        *   `steps: int`: The number of positions to shift the slots. Positive for left rotation, negative for right.
        *   `galois_key: GaloisKey`: A specific Galois key corresponding to the `steps` parameter.
    *   Preconditions:
        *   A `GaloisKey` for the given `steps` (or underlying automorphism index) must have been generated and provided.
        *   The `galois_key` must be compatible with the ciphertext's current level and parameters.
    *   Process:
        1.  **Apply Galois Automorphism**:
            *   Let `ct = (c0, c1)`. An automorphism `ψ_k` is applied to each polynomial component `c0` and `c1`.
            *   If `p(X) = sum p_i X^i`, then `ψ_k(p(X)) = sum p_i X^{i*k} mod (X^N+1)`.
            *   This transforms `ct = (c0(X), c1(X))` which encrypts `m(X)` under `s(X)` into `ct' = (c0(X^k), c1(X^k))`.
            *   This new ciphertext `ct'` encrypts `m(X^k)` but under the secret key `s(X^k)`.
            *   The value `k` is an odd integer from `1` to `2N-1` (specifically, an element of `(Z/2NZ)^*`). The `steps` parameter given by the user needs to be mapped to an appropriate `k`. For CKKS, `k = 5^j mod 2N` often corresponds to slot rotations. The exact mapping from `steps` to `k` depends on the encoding and slot arrangement.
        2.  **Key Switching**:
            *   The ciphertext `ct' = (c0(X^k), c1(X^k))` is now encrypted with respect to `s_rot = s(X^k)`. To make it compatible with other ciphertexts (encrypted with `s(X)`), we need to "key switch" it back to `s(X)`.
            *   The `galois_key` for this rotation `k` (or `steps`) is essentially an encryption of `s(X^k)` under the original secret key `s(X)`.
            *   `galois_key = gk_k = (b_k, a_k)` such that `b_k + a_k * s(X) \approx P * s(X^k)` (or some variant for key switching).
            *   The key switching operation takes `c1(X^k)` (which is the part multiplied by `s(X^k)` in the decryption `c0(X^k) + c1(X^k)s(X^k)`) and transforms it using `gk_k` to be compatible with `s(X)`.
            *   This is similar to relinearization: `(new_c0, new_c1) = KeySwitch( (c0(X^k), c1(X^k)), gk_k, current_s_is_s_of_Xk, target_s_is_s_of_X )`.
            *   Effectively, `c1_prime = c1(X^k)` is treated as the term to be switched. The result of key switching `(switched_c0, switched_c1)` is such that `switched_c0 + switched_c1 * s(X)` decrypts to roughly the same as `c1(X^k) * s(X^k)`.
            *   The new ciphertext is `( c0(X^k) + switched_c0, switched_c1 ) mod q`.
    *   Output: A new `Ciphertext` representing the rotated data, encrypted under the original secret key `s(X)`.
    *   Noise: Noise increases due to the key-switching operation.
    *   Scale: Remains the same as the input `ct.scaling_factor`.
    *   Level: Remains the same as the input `ct.level`.

## 4. Galois Keys (`GaloisKey`, `GaloisKeys`)

*   A `GaloisKeys` object would store a dictionary of `GaloisKey` instances, mapping from `k` (the automorphism index, e.g., specific powers of 5 mod 2N) or `steps` to the actual key material.
*   A single `GaloisKey` (for a specific `k`) typically consists of a vector/list of pairs of polynomials, similar to a `RelinKey`, to handle digit decomposition for better noise performance during key switching.
    *   `gk_k = [ (b_k_i, a_k_i), ... ]`
    *   `b_k_i + a_k_i * s(X) \approx P * g_i * s(X^k)`, where `g_i` are decomposition basis elements.
*   **Generation**: `GaloisKey` for `s(X^k)` is generated by the `KeyGenerator` by encrypting `s(X^k)` under `s(X)`. This involves sampling fresh randomness `a_gk` and error `e_gk`, then computing `b_gk = -a_gk * s(X) + e_gk + P * s(X^k) mod q_P`. (Here `q_P` is a special modulus for keys, or the top-level ciphertext modulus).

## 5. Key Switching Mechanism (`key_switch`)

This is a core subroutine used in rotations and relinearization.
*   **`key_switch(self, input_poly_to_switch: Polynomial, key_for_switching: SpecificRelinOrGaloisKeyComponent, current_q: int) -> (Polynomial, Polynomial)`**:
    *   Input:
        *   `input_poly_to_switch`: The polynomial part of the ciphertext that needs its underlying key changed (e.g., `c1(X^k)` in rotations, or `d2` in relinearization).
        *   `key_for_switching`: The specific key material (e.g., `gk_k` or `rlk`). This key encrypts the "source" key part (e.g., `s(X^k)` or `s^2`) under the "target" key (e.g., `s(X)`).
    *   Process (Simplified for one key component `K = (K_0, K_1)` which encrypts `P * s_source`):
        1.  Decompose `input_poly_to_switch` into digits `d_j` if the key supports it (e.g., `input_poly_to_switch = sum d_j * P_j`). For simplicity, assume no decomposition first.
        2.  The key `K = (K_0, K_1)` is such that `K_0 + K_1 * s_target \approx P * s_source`.
        3.  The switched components are:
            *   `switched_c0 = round(input_poly_to_switch * K_0 / P) mod current_q`
            *   `switched_c1 = round(input_poly_to_switch * K_1 / P) mod current_q`
            (This division by `P` is crucial and implies `P` is part of the key definition or context).
    *   Output: A pair of polynomials `(switched_c0, switched_c1)`.

Applying this to rotation:
`ct' = (c0_automorphed, c1_automorphed)` where `c0_automorphed = c0(X^k)`, `c1_automorphed = c1(X^k)`.
The part to switch is `c1_automorphed`. Let `gk_k` be the Galois key for `s(X^k)`.
`(ks_output_c0, ks_output_c1) = key_switch(c1_automorphed, gk_k, current_q)`
The final rotated ciphertext is:
`rotated_ct_c0 = (c0_automorphed + ks_output_c0) mod current_q`
`rotated_ct_c1 = ks_output_c1 mod current_q`

## 6. Polynomial Automorphism `p(X^k)`

*   This operation involves substituting `X` with `X^k` in the polynomial `p(X) = sum p_i X^i`, resulting in `p(X^k) = sum p_i (X^k)^i`. The exponents are then reduced modulo `N` (since `X^N = -1`).
*   `X^j -> X^{j*k mod 2N}`. The coefficients `p_i` remain the same, but their positions effectively change.
*   Example: `p(X) = p0 + p1*X + p2*X^2`. `k=3`. `N` large.
    `p(X^3) = p0 + p1*X^3 + p2*X^6`.
*   This can be implemented by permuting coefficients based on the powers of `X^k mod (X^N+1)`.

## 7. Mapping `steps` to Automorphism Index `k`

*   For CKKS with `poly_degree = N`, the slots can be thought of as indexed from `0` to `N/2 - 1`.
*   A rotation by `j` slots often corresponds to applying the automorphism `X -> X^k` where `k = g^j mod 2N` for a generator `g` of `(Z/2NZ)^*`. A common generator used is `g=5`.
*   The library will need a predefined mapping or a way to calculate `k` from `steps`. For simplicity, we might initially only support `steps` for which `k` is a precomputed power of a base generator (e.g., `k=5` for 1 step, `k=5^2` for 2 steps, etc., depending on the specific slot ordering).
*   The user provides `steps`, the `Evaluator` looks up the required `k` and corresponding `GaloisKey`.

## 9. Key Considerations

*   **Complexity of Key Switching**: This is one of the more complex parts of FHE. The explanation above is simplified. Real implementations involve decomposing the polynomial being switched into digits/windows to manage noise and key size. Our educational library will use a simplified key switching model.
*   **Galois Key Generation**: Generating all necessary Galois keys can be computationally intensive and result in large key sizes. Typically, only keys for needed rotation amounts are generated.
*   **Noise**: Rotations are noisy operations due to key switching. The parameters must account for this.
*   **Correct `k`**: The choice of `k` for the automorphism `X -> X^k` is crucial and tied to the encoding scheme's slot arrangement. Using `k` values that are powers of a generator `g` (often 5) in `(Z/2NZ)^*` typically generates the desired slot permutations.

This design provides the framework for implementing homomorphic rotations, a key feature for advanced applications of CKKS.
